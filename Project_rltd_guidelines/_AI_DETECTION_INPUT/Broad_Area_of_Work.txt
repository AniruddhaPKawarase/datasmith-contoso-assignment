This dissertation does not sit cleanly inside one academic field. It pulls from five. Multi-agent systems. NLP. Information retrieval. Database systems. Conversational AI. The actual contribution lives in the seam between them, not inside any single one. I'll explain what I mean.

Take a real example. A finance manager at my employer types into a chat box: "why did our Asian fulfilment cost jump last quarter?" The system has to figure out what "fulfilment cost" means here (we have three definitions internally), find the right tables across three databases, do the date math for last quarter, run the SQL, and answer in English. On a real enterprise schema, the best public systems get this kind of question wrong about nine times out of ten. That is the gap I want to close.

So what does each of the five fields actually do for me?

Multi-Agent Systems. One LLM cannot hold the whole picture. So I split the work. Five role-bound agents — Inventory, Logistics, Finance, Demand, Compliance — each writes its own slice of SQL. A Composer stitches them. The interesting bit is the failure path: if execution errors out, the broken fragment goes back to the agent that wrote it, not to a global retry. That feedback shortcut does most of the heavy lifting.

NLP. Standard pieces: intent classification, schema linking, ambiguity detection, temporal expressions, dialogue state. Boring but necessary. One twist — open weights only. Qwen 2.5-Coder is the main model. DeepSeek-Coder and Llama 3.x sit on the bench, depending on what the sub-task needs and what fits in the laptop's VRAM.

Information Retrieval. Six hundred Odoo tables is far too many for any prompt window. So I retrieve. First a schema-aware pass to narrow down. Then a hybrid lexical-and-dense pass to pick the right few-shot examples. Then, when something is genuinely ambiguous and a glossary cannot help, a column-content peek to disambiguate.

Database Systems. Federation across three engines. PostgreSQL for the ERP. DuckDB for the analytics warehouse. Redis for live inventory. One answer can pull from all three. That means dialect translation, automatic currency conversion, and row-level security clauses written into the SQL based on who is asking.

Conversational AI. All of the above runs inside a multi-turn dialogue. Context carries forward. Pronouns resolve. Queries can be narrowed step by step. And, when the user's intent is genuinely under-specified, the system asks back rather than guessing. That last point matters more than people think.
